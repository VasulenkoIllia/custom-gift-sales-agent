# AI Sales Agent MVP Plan (Telegram First)

## 1. Мета проекту
Побудувати AI-агента, який:
- консультує клієнта в месенджері;
- підбирає товари з каталогу;
- веде клієнта до покупки;
- зберігає пам'ять про клієнта;
- робить фолоуапи;
- передає ліда/замовлення в CRM.

Стартовий канал MVP: **Telegram**.

## 2. Режим проекту
Обраний режим: **Standard**.
Причина: є інтеграції (месенджер + CRM), база даних, пам'ять користувача, процес продажу і подальше масштабування на інші канали.

## 3. Scope MVP (що робимо в першій версії)
- Telegram бот з webhook-обробкою повідомлень.
- Єдина база даних для клієнтів, діалогів, товарів, лідів і подій.
- RAG-підбір товарів по вашому каталогу.
- Пам'ять по клієнту (профіль + історія + коротке summary).
- Базовий sales flow: виявлення потреби -> підбір -> робота з запереченнями -> закриття на замовлення.
- Фолоуапи через відкладені задачі.
- Передача ліда/замовлення в CRM через API.
- Мінімальна аналітика: конверсії та якість діалогів.

## 4. Out of Scope MVP (на потім)
- Омніканальність (Instagram/WhatsApp/Facebook/TikTok) в продакшн-рівні.
- Складні персоналізовані кампанії та сегментація на рівні CDP.
- Повноцінний кабінет менеджера з UI (достатньо адмін endpoint + логування).

## 5. Рекомендований стек технологій

### Backend
- **Node.js + TypeScript**
- **Fastify** (webhook/API)
- **Zod** (валідація вхідних payload)
- **Pino** (структуровані логи)

### AI layer
- LLM API (провайдер-агностичний адаптер)
- Embeddings API для пошуку товарів і knowledge base
- RAG orchestration в сервісному шарі (без важких фреймворків на старті)

### Data / Storage
- **PostgreSQL 16 + pgvector** як єдина база для MVP:
  - реляційні дані (users, messages, leads, orders);
  - векторні embeddings (товари, FAQ, summary);
  - JSONB для гнучких полів профілю.
- **Redis + BullMQ** для фолоуапів, ретраїв, відкладених задач.

### Integrations
- **Telegram Bot API** (вхідні/вихідні повідомлення, webhook)
- **CRM API adapter** (upsert lead, create order, sync status)

### Quality
- **Vitest** (unit/integration)
- **Supertest** (API/webhook integration)
- **Docker Compose** (локальний стек: app + postgres + redis)

## 6. Чому одна база = PostgreSQL + pgvector
- Один source of truth для всіх сутностей (клієнт, діалоги, товари, lead/order).
- Менше синхронізацій і розсинхрону між SQL та окремою векторною БД.
- Простий MVP-операційний контур.
- Легко масштабувати: індекси, партиціювання, read replicas.

## 7. Модель даних (MVP)
Основні таблиці:
- `customers` — профіль клієнта, канал, timezone, consent flags.
- `conversations` — сесії діалогів.
- `messages` — вхідні/вихідні повідомлення з метаданими.
- `customer_memory` — слоти пам'яті (потреби, бюджет, уподобання, заперечення).
- `memory_snapshots` — summary по сесії.
- `products` — SKU, ціни, атрибути, наявність.
- `product_embeddings` — embedding товару/опису.
- `knowledge_items` — FAQ/політики/доставка/оплата.
- `knowledge_embeddings` — embeddings для knowledge base.
- `leads` — статус ліда, score, джерело.
- `orders` — оформлені замовлення, CRM external ID.
- `followups` — відкладені дії, час виконання, статус.
- `integration_events` — журнал інтеграційних викликів (idempotency + retry).

## 8. Механізм наповнення новими SKU
Пайплайн оновлення каталогу:
1. Джерело: CSV/Google Sheet/API (визначається вашим процесом).
2. `catalog_import` job:
   - валідація структури;
   - нормалізація атрибутів;
   - upsert в `products`;
   - soft-disable SKU, які зникли з фіда.
3. `embedding_refresh` job:
   - генерація/оновлення embeddings для нових або змінених SKU.
4. Перевірка якості:
   - count імпортованих SKU;
   - count помилок;
   - порівняння diff з попереднім імпортом.

Режим на MVP: cron кожні 1-3 години або manual trigger.

## 9. Логіка роботи агента в діалозі
1. Прийом повідомлення.
2. Нормалізація + збереження в історію.
3. Завантаження customer memory + релевантного контексту (RAG).
4. Визначення етапу продажу (intent/stage).
5. Генерація відповіді в заданому тоні з правилами без галюцинацій.
6. Запис відповіді, оновлення memory, оновлення lead score.
7. За умовою — постановка follow-up або передача в CRM.

## 10. Тон агента (Sales Playbook)
Принципи:
- дружній, професійний, короткі повідомлення;
- спочатку уточнюючі питання, потім підбір;
- не вигадує: відповідає лише на базі каталогу/knowledge base;
- якщо даних нема — чесно каже і пропонує ескалацію;
- завжди веде до next step: "обрати варіант", "підтвердити замовлення", "залишити контакт".

Рекомендований стиль:
- 1-3 абзаци, без "стіни тексту";
- конкретні пропозиції: до 3 товарів за раз;
- для кожного товару: користь, ціна, коротка причина релевантності;
- м'яке закриття: CTA на замовлення.

## 11. Пам'ять по клієнту
Зберігати:
- явні факти: ім'я, місто, мова, бюджет, категорії інтересу;
- поведінкові сигнали: що дивився, що відхиляв, частота контактів;
- поточний sales stage;
- короткий summary попередніх діалогів.

Політика:
- memory оновлюється тільки підтвердженими фактами;
- суперечливі дані маркуються, а не перезаписуються "всліпу";
- критичні дії (замовлення/оплата) тільки через явне підтвердження користувача.

## 12. Follow-up та передача в CRM

### Follow-up rules (MVP)
- якщо замовлення не оформлено -> follow-up через 1 день;
- якщо після першого follow-up замовлення не оформлено -> follow-up через 7 днів;
- повторний інтерес до SKU -> пропозиція альтернативи/апсейлу.

### CRM handoff
Три основні події:
- `lead.created_or_updated`
- `order.draft_created`
- `order.confirmed`

MVP правило:
- `order.confirmed` = клієнт явно підтвердив замовлення, після цього створюємо нове замовлення в KeyCRM.

Вимоги:
- idempotency key на кожну подію;
- retry з backoff;
- журнал інтеграційних помилок;
- двосторонній sync статусу замовлення (опційно для MVP, обов'язково після MVP).

## 13. Етапи реалізації

### Phase 0 — Discovery & Contract (2-4 дні)
- Затвердити CRM контракт (поля lead/order, статуси).
- Затвердити формат товарного фіда.
- Затвердити tone guide.

### Phase 1 — Core Platform (4-6 днів)
- Підняти сервіс (Fastify), PostgreSQL+pgvector, Redis.
- Базова схема БД + міграції.
- Telegram webhook endpoint.

### Phase 2 — Catalog + RAG (5-7 днів)
- Імпорт SKU.
- Embedding pipeline.
- Retrieval + ранжування товарів.

### Phase 3 — Sales Brain + Memory (6-8 днів)
- Sales state machine.
- Memory write/read policy.
- Guardrails проти галюцинацій.

### Phase 4 — CRM + Follow-ups (4-6 днів)
- CRM adapter + idempotency.
- BullMQ jobs для follow-up.
- Retry/alerting механізм.

### Phase 5 — MVP Pilot (7-14 днів)
- Тест на реальних діалогах.
- Збір метрик.
- Корекція промптів/правил/ранжування.

## 14. Критерії успішності MVP
- 95%+ повідомлень обробляються без падінь.
- < 3 сек середній час відповіді (без важких медіа).
- 0 критичних випадків "вигаданих" характеристик товару.
- Конверсія в lead не нижче базової ручної воронки.
- Успішний CRM handoff для 99% підтверджених замовлень.

## 15. План після MVP
- Підключення Instagram/WhatsApp/Facebook через channel adapters.
- Уніфікована омніканальна подієва модель.
- Human handoff інтерфейс для менеджера.
- A/B тестування sales prompt і follow-up сценаріїв.

## 16. Post-MVP: Розділення контекстів в одному чаті
Проблема: в одному Telegram-чаті можуть йти кілька незалежних сценаріїв (консультації, замовлення, повторні звернення). Якщо використовувати лише загальну історію, є ризик змішати контексти.

Рішення: додати окрему сутність `sales_thread` (або `order_draft`) як робочий контекст угоди всередині `conversation`.

Що зберігаємо в `sales_thread`:
- `customer_id`, `conversation_id`, `channel`
- `status`: `open`, `awaiting_confirmation`, `won`, `lost`, `canceled`
- `selected_product_id`, `selected_offer_id`
- `selected_snapshot` (зафіксовані факти на момент підтвердження: назва, ціна, валюта, кількість, атрибути)
- `started_at`, `closed_at`, `last_activity_at`

Операційна логіка:
1. Новий товарний запит -> відкриваємо новий `sales_thread` або переключаємо активний за правилами.
2. Підбір/уточнення -> працюємо в межах активного `sales_thread`.
3. Намір купити -> переводимо `sales_thread` у `awaiting_confirmation`.
4. Явне підтвердження -> створюємо `crm_order` саме з активного `sales_thread`, після чого `sales_thread` -> `won`.
5. Новий незалежний інтерес після `won/lost` -> створюємо новий `sales_thread`.

Наслідки:
- Підтвердження замовлення не змішується з давніми товарами з історії.
- Follow-up прив'язується до конкретної угоди, а не до всього чату.
- Аналітика стає коректною: конверсія, середній цикл, причини втрат по `sales_thread`.
